Section 01

Why Deep Learning for Construction?

Construction is Not a Factory

When people think about applying AI to industry, manufacturing tends to come to mind first — and for good reason. A conveyor-belt assembly line is highly structured: the same parts arrive in the same order, under the same lighting, at the same station, day after day. The environment is controlled, repetitive, and predictable.

A construction site is the opposite. Every project is a one-off. The physical environment changes daily — earthworks give way to foundations, then structural frames, then cladding. Even the same safety hazard looks different at 8 AM in summer sunlight versus 4 PM on an overcast day in winter.

🏗️

This is the core tension: construction sites are perpetually unstructured environments, yet the very strength of deep learning comes from finding patterns in structure. Understanding this gap is the first step toward applying AI meaningfully in the field.

Manufacturing
(conveyor-based)		Construction site
EnvironmentFixed, controlled, indoorsOutdoors, changes daily
Work typeUniform, repeated tasksExcavation → structure → finishing → MEP…
WorkforceStable, trained operatorsRotating subcontractors, mixed skill levels
Visual sceneNear-identical frame to frameLighting, clutter, layout change constantly
Data distributionNarrow and stableWide and continuously shifting

Why This Makes Deep Learning Hard to Apply

The unstructured nature of construction sites creates a fundamental challenge at the data level. In manufacturing, a dataset collected over two weeks may generalize well for years. In construction, a dataset collected at one project type, in one phase, under one set of conditions can fail badly when deployed at a different site.

📊

This does not mean deep learning is the wrong tool — it means we must be deliberate about what we train on, where we deploy, and what we claim the model can do. Domain adaptation, transfer learning, and site-specific fine-tuning are necessities in construction AI.

What Deep Learning Can (and Cannot) Do

Despite these challenges, deep learning remains the most capable tool available for processing the kinds of data construction sites generate. Deep learning discovers features automatically, given sufficient data. The models do not replace expert judgment — they extend its reach across time, scale, and data volume in ways no human team can match alone.

ApproachStrengthLimitation
Rule-based (checklist)Transparent, easy to auditCannot generalize; misses edge cases
Traditional MLWorks well on structured/tabular dataRequires manual feature engineering
Deep LearningHandles images, text, and sequences nativelyNeeds diverse data; hard to generalize across sites
Section 02

What is Deep Learning?

Machine Learning vs. Deep Learning

Machine learning is a broad field where algorithms learn patterns from data rather than following hard-coded rules. Deep learning is a subset that uses neural networks with many layers. Each layer learns progressively more abstract features.

How a Neural Network Works

A neural network is made up of layers of nodes (neurons). Each node receives inputs, multiplies them by weights, adds a bias, and passes the result through an activation function. The key insight is that the network learns which weights matter — you don't tell it what features to look for.

Activation Functions

Without activation functions, stacking layers would be mathematically equivalent to a single linear equation. Activation functions introduce non-linearity, allowing deep networks to model complex real-world patterns. Common choices: ReLU, Sigmoid, and Softmax.

🏗️

In a construction inspection model, the output layer might use Softmax to produce probabilities across classes like safe / minor violation / critical violation. The class with the highest probability becomes the model's prediction.

Section 03

Key Concepts & Math

This section builds the vocabulary you need before diving into specific architectures. We cover the essential math (kept intuitive), the training loop, regularisation strategies, and how each concept translates to a construction-domain problem.

Part 1 · Math Foundations

Vectors, Matrices & Tensors — the language of data

Every piece of data a network processes is ultimately an array of numbers. Understanding the shape of data at each stage is the first practical skill in deep learning — it tells you why certain operations are legal and how information flows through a model.

Loss Functions — measuring the error

Before we can talk about gradients, we need something to differentiate. A loss function L collapses the gap between the model's prediction ŷ and the ground-truth label y into a single number. The entire training process is an optimisation loop that minimises this number.

MSE — Regression
L = (1/n) Σ (yᵢ − ŷᵢ)²
Squared errors penalise large deviations strongly.
Use for: schedule delay prediction, cost estimation.
Cross-Entropy — Classification
L = −Σ yᵢ · log(ŷᵢ)
Punishes confident wrong predictions sharply.
Use for: safety class (safe / minor / critical).
🏗️

For PPE detection on a construction site, use Cross-Entropy. For predicting the number of delay days or final project cost, use MSE (or MAE for reduced sensitivity to outliers).

Partial Derivatives & Gradient — which way is downhill?

Now that we have a loss value L, the question is: which weights should change, and in which direction? A partial derivative ∂L/∂w answers this for a single weight. Stacking all partial derivatives into one vector gives the gradient — an arrow pointing in the steepest uphill direction. We move in the opposite direction to reduce the loss.

Gradient definition
∇wL = [ ∂L/∂w₁, ∂L/∂w₂, …, ∂L/∂wₙ ]
L = loss, w = weight vector. Each component tells you how much that one weight is responsible for the current error.
Part 2 · Training Mechanics

Forward → Loss → Backward → Update

Every iteration of training follows the same four-step cycle.

1
Forward Pass

Input flows layer-by-layer: each layer computes z = W·x + b, then applies a = σ(z). The final layer outputs prediction ŷ.

2
Loss Calculation

Compare ŷ against label y using the chosen loss function. The result is a single number — the error signal for this batch.

3
Backward Pass (Backpropagation)

Using the chain rule, gradients flow from the output back to the input layer. Each weight receives ∂L/∂w.

4
Weight Update

The optimiser nudges every weight: w ← w − η · ∂L/∂w

Part 3 · Training Control

Optimisers — SGD, Momentum & Adam

The optimiser decides how to take the gradient step. Adam is the de-facto default for most construction deep learning tasks.

OptimiserUpdate unitCharacteristicConstruction AI use
Vanilla GDFull datasetStable but extremely slowRarely used in practice
SGDMini-batchFast, can be noisyWith momentum for CNNs
SGD + MomentumMini-batchSmooths oscillations via velocityFine-tuning pre-trained backbones
AdamMini-batchAdaptive per-weight lr, robustDefault for site image models
Adam — core idea
w ← w − η · m̂ / (√v̂ + ε)
m̂ = bias-corrected EMA of gradients (momentum term)
v̂ = bias-corrected EMA of squared gradients (scale normaliser)

Learning Rate & Schedulers

The learning rate η controls how large each update step is. Too large → training diverges; too small → converges slowly. Decaying η over training epochs — using a scheduler — almost always improves final performance.

Epoch / Batch / Iteration

These three terms describe how much data the model processes at different granularities. Getting them confused is one of the most common sources of miscommunication when reading papers or debugging code.

Overfitting & Underfitting

Overfitting occurs when the model memorises training data rather than generalising. The validation loss curve is the primary diagnostic — not training loss alone.

Train / Validation / Test Split

Splitting into three distinct sets allows fair hyperparameter tuning and an unbiased final performance report. In construction, always split by project, not by image — mixing images from the same site across train and test is a data leakage problem that inflates reported metrics.

Dropout & Regularisation

Dropout randomly deactivates a fraction p of neurons during each training step. L2 regularisation adds a weight-magnitude penalty to the loss.

Dropout
Each neuron: off with prob p
At inference: all neurons on, outputs scaled by (1 − p).
Typical p = 0.3–0.5.
L2 Regularisation
Ltotal = L + λ · Σ wᵢ²
λ controls penalty strength. Equivalent to weight decay: w ← w · (1 − λη)
Part 4 · Construction Domain Connections

Transfer Learning

A model pre-trained on ImageNet has already learned to detect edges, textures, and shapes. Fine-tuning uses this as a starting point — freezing the backbone and training only a new task-specific head on a small construction dataset.

Data Augmentation

Construction images vary dramatically in lighting, camera angle, and weather. Augmentation artificially diversifies the training set by applying random transforms at training time.

Class Imbalance

In safety monitoring datasets, violation images can account for fewer than 5% of all frames. A naive model that always predicts "safe" achieves 95% accuracy — yet is completely useless in practice.

⚠️

Never rely on accuracy alone in imbalanced settings. Always report Precision, Recall, and F1-score per class. In safety monitoring, a missed detection (false negative) is typically far more costly than a false alarm.

StrategyHow it worksConstruction application
OversamplingDuplicate minority-class samples, or synthesise via SMOTEAugment violation images before retraining
Class WeightsAssign higher loss weight to the minority classw = Nsafe / Nviolation
Focal LossDown-weights easy samples, focusing on hard onesEffective for rare or ambiguous site violations
Threshold TuningLower the classification threshold below 0.5Prefer false alarm over missed hazard in live monitoring
✅

Combining transfer learning + data augmentation + class weights is the most practical starting point for construction vision models when you have only a few hundred labelled images.

Section 04

CNN — Seeing the Site

Why CNNs? The Problem with Flattening Pixels

Construction sites produce an enormous volume of visual data — CCTV feeds, drone footage, on-helmet cameras. A standard fully-connected network processes images by flattening them into a 1D vector, which destroys all spatial relationships between pixels. A Convolutional Neural Network (CNN) preserves that 2D structure: it slides small learned filters across the image, detecting edges, textures, and shapes wherever they appear — regardless of position.

This property, called translation invariance, is exactly what a construction site demands. A hardhat is a hardhat whether it sits in the top-left corner of the frame or the bottom-right, in bright noon light or overcast shadow.

YOLOv8 Architecture: Backbone → Neck → Head

Object detection models are split into three functional stages. The Backbone extracts hierarchical features from the image. The Neck fuses features across multiple scales — critical when you need to detect a distant worker (small object) and a nearby excavator (large object) in the same frame. The Head outputs the final bounding box coordinates and class probabilities.

ComponentRoleWhy it matters on-site
Backbone (CSPDarknet)Hierarchical feature extractionLearns helmet shape, texture, and color invariantly
Neck (PANet / FPN)Multi-scale feature fusionDetects far workers and near machinery simultaneously
Head (Decoupled)BBox regression + classificationOutputs pixel location + PPE compliance probability
Transfer LearningReuses ImageNet-pretrained weightsConverges fast even with limited site-specific data
💡

Transfer Learning in one sentence: the backbone's early layers already know how to detect edges, textures, and shapes from ImageNet's 1.2 M images — we only fine-tune the upper layers on construction site data.

Section 05

NLP — Reading the Documents

If CNN is the model that sees the site, NLP is the model that reads it. Every construction project generates an enormous volume of text: RFIs, incident reports, daily logs, contract clauses, meeting minutes. These documents contain critical information that is currently processed manually — slowly, inconsistently, and at scale impossible for any human team to sustain. Natural Language Processing changes that.

📄

A mid-size building project generates thousands of RFIs, hundreds of incident reports, and tens of thousands of email threads over its lifetime. NLP models can read, classify, and extract structured information from all of it — automatically.

Part 1 · Why Text is Hard for Computers

Images have a clear numeric structure — pixels on a 2D grid. Text does not. Words carry meaning only in relation to other words, and the same word can mean entirely different things depending on context. Construction language compounds this with dense abbreviations, code numbers, and domain-specific jargon that general-purpose models have never seen.

ChallengeGeneral languageConstruction domain
Ambiguity"bank" = river bank or financial bank"load" = structural load, truck load, or software load
AbbreviationsCommon (e.g. USA, CEO)Dense: RFI, NCR, PCO, MEP, ASTM A615 Gr.60
Format variationSentences follow grammar rules"weather hold – ext pour suspended 5d" is valid
Vocabulary size~170 k English wordsSpec numbers, grid codes, equipment models not in dict

Part 2 · Tokenisation & Embeddings

Before a model can process text, it must convert words into numbers. This happens in two stages: tokenisation splits text into manageable units, and embeddings map those units to dense numeric vectors that capture meaning.

Embeddings convert each token into a dense vector where semantically similar tokens are close in space. Early methods like Word2Vec use a single fixed vector per word. Modern contextual embeddings (BERT, etc.) produce a different vector for the same word depending on the surrounding context — critical for construction terms that are inherently polysemous.

Part 3 · Key NLP Architectures

Three architectural generations underpin modern construction NLP. Each solved a problem the previous one could not.

The Transformer removes recurrence entirely. Every token attends to every other token in parallel, which makes training dramatically faster and allows the model to capture long-range dependencies — essential for a clause at the start of a 200-page contract that qualifies a clause at the end.

G2
Fine-tuned BERT for RFI classification
Train bert-base-uncased on 25 labelled RFIs (5 per class). Uses stratified split. ~1 min on T4 GPU.
PythonLab G2 · Fine-tune BERT
import torch, random
from collections import defaultdict
from torch.utils.data import Dataset, DataLoader
from transformers import (AutoTokenizer, AutoModelForSequenceClassification,
                            AdamW, get_linear_schedule_with_warmup)
from sklearn.metrics import classification_report

LABEL2ID = {"design":0, "scope":1, "schedule":2, "quality":3, "safety":4}
ID2LABEL = {v: k for k, v in LABEL2ID.items()}

raw_data = [
    # design (5)
    ("Clarify shear wall reinforcement at grid B-4 per drawing S-22.", "design"),
    ("Column footing detail conflicts with architectural plan on floor 1.", "design"),
    ("Staircase handrail height does not comply with IBC 1014.2.", "design"),
    ("Curtain wall anchor spacing differs between MEP and structural drawings.", "design"),
    ("Confirm fire-rated assembly spec for corridor wall type W-3.", "design"),
    # scope (5)
    ("Owner requests additional electrical outlets in conference rooms.", "scope"),
    ("Added scope for roof drainage upgrade not in original contract.", "scope"),
    ("Client wants security camera coverage extended to parking level B2.", "scope"),
    ("Landscape scope expanded to include irrigation system on east plot.", "scope"),
    ("Fit-out of lobby expanded per revised architectural program.", "scope"),
    # schedule (5)
    ("Structural steel fabrication delayed 3 weeks; revised delivery date TBD.", "schedule"),
    ("Weather hold suspended exterior concrete pours for 5 days.", "schedule"),
    ("MEP rough-in on floor 8 cannot start until rebar inspection clears.", "schedule"),
    ("Subcontractor manpower shortage pushing tile installation by 2 weeks.", "schedule"),
    ("Permit approval from authority delayed; foundation work on hold.", "schedule"),
    # quality (5)
    ("Concrete cylinder test at 28 days failed to achieve 4,000 psi design strength.", "quality"),
    ("Weld inspection identified incomplete fusion on column splice at level 5.", "quality"),
    ("Waterproofing membrane shows blistering on podium deck area C.", "quality"),
    ("Formwork deflection exceeded allowable tolerance during pour.", "quality"),
    ("Rebar spacing on slab grid 3-5 deviates from approved shop drawing.", "quality"),
    # safety (5)
    ("Missing guardrail on scaffold west face creates fall hazard above 6 m.", "safety"),
    ("Excavation edge near crane path lacks adequate shoring; collapse risk.", "safety"),
    ("Workers observed not wearing PPE in designated hard-hat zone.", "safety"),
    ("Temporary power cables on floor 3 are not properly secured; trip hazard.", "safety"),
    ("Hot-work permit not issued before welding commenced in level B1.", "safety"),
]

random.seed(42)
by_class = defaultdict(list)
for text, label in raw_data:
    by_class[label].append((text, label))

train_data, val_data = [], []
for label, samples in by_class.items():
    random.shuffle(samples)
    val_data.append(samples[0])
    train_data.extend(samples[1:])
random.shuffle(train_data)

MODEL_NAME = "bert-base-uncased"
tokenizer  = AutoTokenizer.from_pretrained(MODEL_NAME)

class RFIDataset(Dataset):
    def __init__(self, data):
        self.enc = tokenizer([d[0] for d in data],
                             truncation=True, padding=True,
                             max_length=128, return_tensors="pt")
        self.labels = torch.tensor([LABEL2ID[d[1]] for d in data])
    def __len__(self):  return len(self.labels)
    def __getitem__(self, i):
        return {k: v[i] for k, v in self.enc.items()}, self.labels[i]

train_loader = DataLoader(RFIDataset(train_data), batch_size=8, shuffle=True)
val_loader   = DataLoader(RFIDataset(val_data),   batch_size=8)

model = AutoModelForSequenceClassification.from_pretrained(
    MODEL_NAME, num_labels=5, id2label=ID2LABEL, label2id=LABEL2ID).to(device)
EPOCHS    = 5
optimizer = AdamW(model.parameters(), lr=2e-5, weight_decay=0.01)
scheduler = get_linear_schedule_with_warmup(
    optimizer,
    num_warmup_steps=len(train_loader) * EPOCHS // 10,
    num_training_steps=len(train_loader) * EPOCHS)

for epoch in range(EPOCHS):
    model.train(); total_loss = 0
    for batch, labels in train_loader:
        batch   = {k: v.to(device) for k, v in batch.items()}
        labels  = labels.to(device)
        outputs = model(**batch, labels=labels)
        outputs.loss.backward()
        optimizer.step(); scheduler.step(); optimizer.zero_grad()
        total_loss += outputs.loss.item()
    print(f"Epoch {epoch+1}/{EPOCHS}  loss={total_loss/len(train_loader):.4f}")

model.eval(); preds, trues = [], []
with torch.no_grad():
    for batch, labels in val_loader:
        logits = model(**{k: v.to(device) for k, v in batch.items()}).logits
        preds.extend(logits.argmax(-1).cpu().tolist())
        trues.extend(labels.tolist())

print(classification_report(trues, preds,
      labels=list(LABEL2ID.values()),
      target_names=list(LABEL2ID.keys()), zero_division=0))

    ("4.2", "Reinforcing steel shall conform to ASTM A615 Grade 60. Mill certificates must be submitted prior to delivery."),

    ("5.2", "Defective work identified during inspection shall be corrected within 7 days at no additional cost to the Owner."),

        i = idx.item()
        print(f"  Clause {clause_ids[i]}  |  score={score:.3f}")
        print(f"  {clause_texts[i]}\n")

Section 06

LLM & RAG — Talking to Knowledge

The Document Problem in Construction

A typical construction project generates thousands of pages of unstructured text — RFIs, safety reports, specifications, contracts, inspection logs. The information is all there, but finding the right clause in a 300-page specification at 7 AM on-site is a different problem from image classification. Large Language Models (LLMs) are the most capable tool we have for processing this kind of data — but used naively, they hallucinate. Retrieval-Augmented Generation (RAG) grounds the model's answers in your actual documents.

ApproachHow it worksRisk
Naive LLMAsk the model directly — it answers from training memoryHallucination, outdated info, no source citation
Fine-tuningRetrain the model on your documentsExpensive, static — needs retraining when docs change
RAGRetrieve relevant chunks → feed as context → generateRetrieval quality matters; slower than direct query

How an LLM Processes Text

An LLM doesn't read words — it reads tokens (roughly word fragments). Each token is converted to an embedding vector, then processed through many Transformer layers. The key mechanism is self-attention: every token can directly attend to every other token in the context window, capturing long-range relationships that RNNs struggled with.

💡

Context window is the maximum number of tokens an LLM can process at once. Gemini 2.5 Flash supports up to 1 million tokens — roughly 700,000 words. Even so, RAG is still preferable for long document collections: it's faster, cheaper, and forces the model to cite its sources.

The RAG Pipeline

RAG has two phases. Indexing (done once) converts your documents into searchable vectors. Querying (done at runtime) retrieves the most relevant chunks and passes them as context to the LLM — so the model answers from your documents, not from training memory.

Key Concepts Before the Code

ConceptWhat it isConstruction analogy
TokenSmallest unit of text the model processes (~¾ of a word)A word fragment in a specification clause
EmbeddingDense vector representing the meaning of a text chunkA GPS coordinate for a document's meaning
ChunkFixed-length text segment with optional overlapOne specification article or safety clause

Parse & chunk the document
Split into 800-char chunks with 100-char overlap. Overlap prevents sentences from being cut at chunk boundaries.
PythonStep 3 · Parse & chunk
from pypdf import PdfReader

def load_pdf(path):
    reader = PdfReader(path)
    pages  = []
    for i, page in enumerate(reader.pages):
        text = page.extract_text()
        if text and text.strip():
            pages.append({"page": i + 1, "text": text.strip()})
    return pages

def chunk_pages(pages, chunk_size=800, overlap=100):
    chunks = []
    for p in pages:
        text, pg = p["text"], p["page"]
        start = 0
        while start < len(text):
            end   = min(start + chunk_size, len(text))
            chunk = text[start:end].strip()
            if len(chunk) > 50:
                chunks.append({
                    "id"  : f"p{pg}_c{len(chunks)}",
                    "text": chunk,
                    "meta": {"page": pg, "source": os.path.basename(path)}
                })
            start += chunk_size - overlap
    return chunks

pages  = load_pdf(PDF_PATH)
chunks = chunk_pages(pages)
print(f"📄 Pages : {len(pages)}")
print(f"🧩 Chunks: {len(chunks)}")

def retrieve(query, top_k=3):
    q_emb = embedder.encode([query]).tolist()
    res   = collection.query(
        query_embeddings=q_emb, n_results=top_k,
        include=["documents", "metadatas", "distances"]
    )
    return [
        {"text": d, "page": m["page"], "sim": round(1-s, 3)}
        for d, m, s in zip(
            res["documents"][0], res["metadatas"][0], res["distances"][0]
        )
    ]

def rag_answer(query, top_k=4):
    hits    = retrieve(query, top_k=top_k)
    context = "\n\n---\n\n".join(
        f"[Page {h['page']}]\n{h['text']}" for h in hits
    )
    prompt = f"""You are a construction safety expert.
Answer using ONLY the document excerpts below.
If the answer is not in the excerpts, say so clearly.
Always cite the page number(s) you used.

=== EXCERPTS ===
{context}

=== QUESTION ===
{query}

=== ANSWER ===""
    answer = model.generate_content(prompt).text
    pages  = [h["page"] for h in hits]
    print(f"🔍 Query : {query}")
    print(f"📄 Pages retrieved: {pages}")
    print(f"\n💬 Answer:\n{answer}\n")
    print("─" * 60)
    return answer

for q in [
    "What height requires fall protection for construction workers?",
    "What types of fall protection systems are acceptable?",
    "What are employer responsibilities for fall protection training?",
]:
    rag_answer(q)

Section 07

Fine-tuning — Teaching the Model Your Domain

Section 6 (RAG) retrieves the right document and hands it to the model as context. Fine-tuning does something different: it rewires the model's weights so that construction safety language, terminology, and reasoning patterns become part of what the model is — not just what it reads at inference time. The two approaches are not competing; in practice, the most capable systems combine both.

🏗️

We use Qwen3-1.7B via unsloth — 4-bit quantised, ~2 GB VRAM — fine-tuned on the DBCMLAB/Constructionsafety_QApairs dataset: 5,114 construction safety QA pairs derived from KOSHA guidelines. LoRA rank-16 keeps trainable parameters at ~1% of the total. Estimated training time ~6 hours on a free Colab T4 GPU.

Part 1 · RAG vs. Fine-tuning — When to Choose Which

Neither approach is universally better. The right choice depends on what you need the model to change: its knowledge (what it knows) or its behaviour (how it communicates and reasons).

RAGFine-tuning
What it changesKnowledge at inference timeModel weights — permanent
Updating knowledgeSwap the document storeRequires re-training
Domain language & toneModel still speaks genericallyFully absorbed into model
Data neededRaw documents (no labels)Thousands of QA pairs
Compute costLow — retrieval + inferenceMedium — training upfront
Hallucination riskLower — grounded in retrieved textHigher — model generates freely
Best for constructionContract Q&A, regulation lookupSafety report generation, terminology

Part 2 · Full Fine-tuning vs. LoRA vs. QLoRA

Fine-tuning a 2.4B model by updating all parameters requires ~20 GB of VRAM — well beyond a free Colab T4. Parameter-Efficient Fine-Tuning (PEFT) methods solve this by learning only a tiny fraction of new weights, leaving the original model frozen.

MethodTrainable paramsVRAM (2.4B)Colab T4
Full fine-tuning100% — all weights~20 GB✗ Not possible
LoRA (bf16)~0.5%~10 GB△ Tight
QLoRA (4-bit + LoRA)~0.5%~6 GB✓ Comfortable
Prompt Tuning< 0.1%~5 GB✓ But weaker

Part 3 · Dataset — KOSHA Construction Safety QA

The DBCMLAB/Constructionsafety_QApairs dataset contains 5,114 Korean QA pairs generated from KOSHA (한국산업안전보건공단) construction safety guidelines using GPT-3.5-turbo. Topics include earth anchoring, excavation, shoring, pipeline work, PPE compliance, and equipment safety — exactly the vocabulary a general LLM lacks.

⚠️

This dataset is synthetically generated by GPT-3.5-turbo, not written by domain experts. It is excellent for demonstrating fine-tuning mechanics and learning construction vocabulary, but do not use the resulting model for real safety decisions without human expert verification.

def ask(question: str, max_new_tokens: int = 512):
    messages = [
        {"role": "system",
         "content": "You are a construction safety expert. Answer questions accurately and in detail."},
        {"role": "user", "content": question},
    ]
    input_ids = tokenizer.apply_chat_template(
        messages,
        tokenize              = True,
        add_generation_prompt = True,
        return_tensors        = "pt",
    ).to("cuda")

    outputs = model.generate(
        input_ids,
        max_new_tokens = max_new_tokens,
        temperature    = 0.7,
        top_p          = 0.9,
        do_sample      = True,
    )
    decoded = tokenizer.decode(
        outputs[0][input_ids.shape[-1]:], skip_special_tokens=True
    )
    return decoded
